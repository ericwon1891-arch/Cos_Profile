# Hero 이미지 레터박스 + 모바일 햄버거 네비게이션 Design

**Goal:** 두 가지 반응형 결함을 고친다 — (1) Hero 배경 사진이 초광폭 데스크톱(예: 2560×1440)에서 위아래가 과도하게 잘리는 문제, (2) 모바일 세로모드에서 네비게이션 항목(현재 5개 이상, 캐릭터 섹션이 늘어날수록 더 늘어남)이 한 줄에 다 안 들어가 정렬이 깨지는 문제.

**배경(근본 원인, systematic-debugging으로 확인됨):**
- `HeroSection.jsx`가 `min-h-screen`(항상 뷰포트 높이 100%)인 섹션 안에 `object-cover` 이미지를 채우는데, 뷰포트 가로세로비가 이미지의 권장 비율(1920×1600 ≈ 1.2)에서 크게 벗어날수록(초광폭 데스크톱, 세로로 긴 모바일) 잘리는 양이 통제 불능으로 커진다. `object-top`/`object-center`는 잘리는 위치만 바꿀 뿐 잘리는 양을 줄이지 못한다.
- `Navbar.jsx`는 반응형 처리가 전혀 없는 한 줄 `flex` 레이아웃이었고, 최근 캐릭터 섹션마다 네비 항목이 동적으로 추가되는 기능이 들어오면서 항목 수가 늘어나 좁은 화면에서 넘치게 됐다.

**Non-goals:** Hero 사진을 여러 장(디바이스별로 다르게 크롭된 파일)으로 관리하는 art-direction 방식은 하지 않는다 — 관리자는 지금처럼 사진 한 장만 올린다. 모바일 메뉴 열림 중 배경 스크롤 잠금은 하지 않는다 (원페이지 스크롤 네비게이션 특성상 불필요).

## 1. Hero 이미지 — 비율 기반 레터박스

- 일반적인 뷰포트 비율(0.75 ~ 1.5 사이, 즉 세로로 심하게 길거나 가로로 심하게 넓지 않은 경우)에서는 지금과 동일하게 `object-cover` + `object-top`(모바일)/`object-center`(데스크톱)를 그대로 사용한다 — 시각적 변화 없음.
- 뷰포트 가로세로비가 1.5보다 크면(초광폭 데스크톱, 예: 2560×1440 ≈ 1.9) 또는 0.75보다 작으면(세로로 긴 모바일) `object-fit: contain`으로 전환한다 — 이미지가 원본 비율 그대로 표시되어 인물이 잘리지 않고, 남는 여백은 섹션에 이미 있는 배경 그라데이션(`bg-gradient-to-b from-[#0a0a0a] to-[#1a1a2e]`)이 자연스럽게 채운다.
- 구현은 Tailwind의 임의 값(arbitrary value) 미디어쿼리 문법(`[@media(min-aspect-ratio:3/2)]:object-contain`, `[@media(max-aspect-ratio:3/4)]:object-contain`)으로 처리한다 — 뷰포트 **너비**가 아니라 **가로세로비** 기준이라 창을 수동으로 리사이즈해도 일관되게 동작하고, JS 리스너나 신규 의존성이 필요 없다.

## 2. Navbar — 모바일 세로모드 햄버거 메뉴

- `md`(768px) 이상: 지금 있는 가로 메뉴를 그대로 유지 (`hidden md:flex`로 감싸서 이 구간에서는 변경 없음).
- `md` 미만: 로고 옆에 햄버거 아이콘 버튼만 보이고(`md:hidden`), 클릭 시 네비바 바로 아래 전체 폭 드롭다운 패널이 펼쳐진다. 항목은 세로로 나열되고 배경은 `bg-black/90 backdrop-blur-sm`로 반투명 처리해 어느 섹션 위에서 열려도 가독성을 확보한다.
- 열림/닫힘 상태는 컴포넌트 로컬 `useState`(`menuOpen`)로 관리한다.
- 항목 클릭 시 해당 섹션으로 스크롤 이동(`react-scroll`의 기존 `Link` 그대로 사용)하면서 `menuOpen`을 `false`로 되돌려 메뉴를 자동으로 닫는다.
- 캐릭터 섹션이 늘어나 항목이 많아져도 세로 리스트라 레이아웃이 깨지지 않는다 (항목이 아주 많아질 경우 패널 자체가 세로로 길어지는 것은 허용 — 페이지가 스크롤되므로 문제 없음).

## 에러 처리

별도 처리 없음 — `data.sections`가 비어 있으면(로딩 중/데이터 없음) 지금과 동일하게 캐릭터 관련 네비 항목만 없고 나머지는 정상 동작.

## 작업 브랜치

이 작업은 `main`에서 분기한 `반응형` 브랜치 위에서 진행한다 — 완료 후 병합 여부는 별도로 결정하며, 필요 시 지금(변경 전) 상태로 언제든 되돌릴 수 있다.

## 테스트

- `HeroSection.jsx`: 가로세로비에 따른 `object-fit` 전환은 순수 CSS 미디어쿼리라 jsdom 기반 유닛 테스트로 직접 검증하기 어렵다 — 클래스 문자열에 해당 미디어쿼리 variant 클래스가 포함되어 있는지 정도만 스냅샷성으로 확인하고, 실제 시각 검증은 브라우저 수동 확인으로 대체한다.
- `Navbar.test.jsx`에 다음 케이스를 추가한다:
  - 햄버거 버튼이 렌더링된다
  - 햄버거 버튼 클릭 시 모바일 메뉴 패널이 열리고 네비 항목들이 보인다
  - 메뉴가 열린 상태에서 네비 항목을 클릭하면 메뉴가 닫힌다 (`menuOpen`이 `false`로 바뀌는지, 패널이 사라지는지)
  - 기존 데스크톱 전용 네비 관련 테스트(항목 존재 여부 등)는 그대로 유지 — `hidden md:flex`/`md:hidden`은 CSS로만 처리되므로 jsdom에서는 두 메뉴 마크업이 모두 DOM에 존재할 수 있음. 이 경우 데스크톱 메뉴와 모바일 메뉴 각각에 별도 테스트 식별자(`data-testid="desktop-nav"`/`data-testid="mobile-nav-panel"`)를 부여해 테스트가 서로 헷갈리지 않게 한다
